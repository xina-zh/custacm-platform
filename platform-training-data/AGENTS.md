# platform-training-data Agent Notes

This module owns the training-data warehouse storage model.

Current state:

- Maven parent module with shared app helper module `training-data-common`, independent OJ modules `training-data-codeforces` and `training-data-atcoder`, plus runnable `training-data-web`.
- Implemented Codeforces storage targets are `ods_codeforces__submission`, `dwd_codeforces__submission`, `dwm_codeforces__handle_problem_first_accepted`, and `dws_codeforces__handle_daily_rating_accepted_summary`. Implemented AtCoder storage targets are `ods_atcoder__submission`, `ods_atcoder__problem`, `ods_atcoder__problem_model`, `dwd_atcoder__submission`, `dwm_atcoder__handle_problem_first_accepted`, and `dws_atcoder__handle_daily_rating_accepted_summary`; AtCoder has Kenkoooo source clients, ODS parsers/writers, recent-window submission collection, startup and low-frequency problem metadata ODS collection, ODS-to-DWD/DWM/DWS SQL task resources, and a warehouse refresh handler for collection jobs and scheduled collection. Public DWD/DWM/DWS read-side queries use the common same-layer table naming contract `dwd_{oj}__submission`, `dwm_{oj}__handle_problem_first_accepted`, and `dws_{oj}__handle_daily_rating_accepted_summary`; additional OJ queries require matching same-structure DWD/DWM/DWS tables before use.
- Implemented OJ handle-account storage target is `oj_handle_account`, keyed by `student_identity`; `handles_json` stores an uppercase OJ-name to handle map such as `CODEFORCES` or `ATCODER`, `need_collect` is the current automatic-collection eligibility flag used together with the schedule `ojName`, and `collection_states_json` stores per-OJ `historyStartReached` plus collector execution `lastCollectedAt`.
- `training-data-common` owns generic OJ app/domain/infra/web code: OJ handle-account use cases, HTTP endpoints, and JDBC repository; DWD/DWM/DWS query services, HTTP endpoints, and repositories; student-data purge orchestration and HTTP endpoint; recent-window submission collection orchestration and HTTP endpoints; OJ collector dispatch; OJ warehouse refresh dispatch; collection handle resolver/adapter contracts, in-JVM overlap skipping, source request retry/rate limiting, sorted-page window filtering, in-process collection job service, aggregate collection result models, scheduled-collection configuration, common same-layer DWD/DWM/DWS table DDL migrations, and tests. It must not own OJ-specific payloads, OJ-specific ODS records/writers, OJ-specific ODS table migrations, source clients, or OJ-specific source parsing.
- Codeforces owns its own ingest application service, Codeforces adapter for recent-lookback submission collection, Codeforces warehouse-refresh handler wiring, collect batch type, ODS record, parser, writer, Codeforces ODS purge adapter, fixture, DDL/Flyway migrations, ODS upsert SQL, DWD/DWM/DWS SQL task resources, SQL task manifest, Spring config, and tests. Codeforces collection fetches one `user.status?handle=...` page at a time; team submissions are attributed to the collected handle when that handle appears in `author.members`, so ODS and DWD use `submission + handle` grain.
- AtCoder owns its ODS Flyway migrations, Kenkoooo source client, submission/problem/problem-model parsers, submission/problem/problem-model ODS writers, recent-window submission collector, startup and low-frequency problem metadata collector, AtCoder ODS purge adapter, AtCoder warehouse-refresh service/handler, and AtCoder DWD/DWM/DWS SQL task resources in `training-data-atcoder`; AtCoder DWD/DWM/DWS physical table DDL lives in `training-data-common` because it follows the common same-layer warehouse contract.
- `training-data-web` exposes module health/info, Codeforces-compatible admin recent-lookback collection by `studentIdentity` with optional `ojName` for Codeforces or AtCoder, Codeforces admin collection-job start/list endpoints, startup AtCoder problem metadata bootstrap, OJ handle-account write endpoints, admin student-data purge with required `ojName`, the public full OJ handle account map endpoint including per-OJ collection states, and public DWD/DWM/DWS query endpoints that keep the Codeforces path and accept `ojName` as the OJ pass-through parameter.
- Manual ODS batch-upsert and standalone warehouse refresh HTTP endpoints are intentionally not exposed. ODS writes happen through collection app services; warehouse refresh can run as a collection-job internal callback when requested and also runs after enabled scheduled collection when the collection produces a batch.
- OJ handle-account creation and identity/collection-flag updates require the platform `admin` role. The public full OJ handle account map endpoint is a guest endpoint and must not parse JWTs.
- Student-data purge requires the platform `admin` role, requires `ojName`, resolves that OJ handle from OJ handle-account data, deletes ODS through the OJ-specific purge adapter and DWD/DWM/DWS through the common warehouse purge adapter in one transaction, keeps the OJ handle binding, and must not delete auth accounts directly.
- HTTP authorization follows [../docs/authorization.md](../docs/authorization.md): `/admin/**` is admin-only, `/player/**` is player/admin, and guest endpoints must not parse JWTs.
- `training-data-web` includes MySQL runtime driver support and applies `classpath:db/migration` scripts with Flyway.
- `training-data-web` must use the project file logging contract from [../docs/logging.md](../docs/logging.md).
- Submission warehouse table contracts, SQL task order, internal collection-job refresh semantics, and source-access notes live in [docs/ods-submission.md](docs/ods-submission.md). AtCoder/Kenkoooo collection design lives in [docs/atcoder-collection.md](docs/atcoder-collection.md).

Rules:

- Keep the current slice focused on Codeforces/AtCoder warehouse modeling, AtCoder Kenkoooo ODS collection including startup problem metadata bootstrap, common OJ handle-account mapping, recent-lookback submission collection, student-data cleanup, OJ-owned SQL task resources, and OJ SQL task refresh paths triggered by collection jobs or enabled scheduled collection.
- Do not add persistent pipeline run state, cross-OJ DAG orchestration, or ADS physical tables until there is a concrete downstream query or product workflow. OJ scheduled collection is an enabled-by-default Spring trigger that calls the same collection use case with the configured schedule `ojName` and refreshes the selected OJ warehouse when a batch is produced; it must not grow into a pipeline scheduler. The current collection job list is process-local status for browser polling only; do not extend it to ODS import or manual warehouse refresh without an explicit product decision.
- Keep each OJ as a vertical Maven module. Shared training-data modules may own reusable OJ app/domain/infra/web logic and common same-layer DWD/DWM/DWS table DDL, but must not own OJ ODS payload/storage contracts or OJ-specific source integrations.
- Keep common HTTP controllers thin; orchestration belongs in common app services or OJ-owned app services reached through common contracts. Do not re-add a manual ODS import controller, standalone warehouse refresh controller, or single collection-job detail endpoint unless the product decision changes.
- OJ handle-account identity migration changes the `student_identity` key in `oj_handle_account` and may update `need_collect`; it does not modify auth accounts and must not change the stored handles map.
- Do not add a unified `OdsSubmissionRecord`, `OdsSubmissionWriter`, or `SourcePlatform` for cross-OJ submission storage. Add OJ-specific records, parsers, writers, fixtures, SQL, and tests instead.
- Do not add ADS physical tables or unified cross-OJ DWD/DWM/DWS tables until there is a real downstream query or product workflow. Same-layer per-OJ DWD/DWM/DWS tables may share the common warehouse query contract.
- Keep Codeforces DWD/DWM/DWS transforms as idempotent SQL resources. Do not add Java row-by-row transformation logic.
- Test external-source parsing and collection with local fixtures or fake clients. Do not make default tests depend on live Codeforces availability.
- When changing module boundaries or HTTP behavior, update [../docs/architecture.md](../docs/architecture.md), [../docs/api.md](../docs/api.md), [../docs/agent/context-map.md](../docs/agent/context-map.md), and [../docs/doc-sync-map.tsv](../docs/doc-sync-map.tsv) if routing rules change.
