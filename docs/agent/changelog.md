# 更新日志写作规则

`CHANGELOG.md` 由 agent 更新，但读者是人。它应该像项目发布说明一样说明成果和影响，不应该写成内部任务流水账。

## 何时更新

每次准备提交 MR 前，都要更新 [../../CHANGELOG.md](../../CHANGELOG.md)。

文档、构建、CI、部署、API、模块边界和业务代码变更都应该留下简短记录。如果某次变更刻意不影响使用者和维护者，在 MR 描述里说明即可，不要为了凑记录写噪音。

## 写入位置

新记录追加到 `## 未发布` 小节最上方，按时间倒序排列。

不要主动创建版本号或发布日期小节，除非用户明确要求整理发布说明。

## 固定格式

使用这个格式：

```md
### YYYY-MM-DD - 短标题

- 成果：用一两句话说明这次 MR 交付了什么。
- 影响：说明对开发者、部署者、使用者或后续 agent 的影响；没有外部影响时写“仅文档/内部维护”。
- 验证：列出实际运行过的验证命令；没有运行时说明原因。
```

## 写作风格

- 面向六个月后快速翻项目历史的维护者写。
- 使用自然中文和具体名词。
- 优先写业务或工程成果，不要罗列文件清单。
- 每条记录控制在三到五个要点内。
- 兼容性、部署、身份模型等重要影响必须明确写出。
- 不写 agent 执行过程、临时尝试、原始 diff 或完整文件列表。
- 没有实际运行的验证命令不能写成已验证。
